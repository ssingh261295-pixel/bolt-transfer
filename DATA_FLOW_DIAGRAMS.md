## Summary

### Key Architectural Patterns

1. **Database as Source of Truth**
   - All persistent state in Postgres
   - UI loads from DB, not API
   - API calls update DB, DB updates UI

2. **Realtime for Propagation**
   - postgres_changes events
   - Filtered by user_id
   - Patch local state, don't reload

3. **Cache for Performance**
   - dashboard_metrics_cache for aggregates
   - gtt_orders cache for Zerodha GTTs
   - Load instantly, sync in background

4. **Edge Functions for Integration**
   - Proxy to Zerodha API
   - Handle auth & RLS
   - Update database after API calls
   - No business logic in client

5. **Server-Side Execution**
   - HMT engine runs 24/7
   - Browser-independent
   - Sub-100ms trigger latency
   - WebSocket to Zerodha

### Performance Characteristics

| Operation | Latency | Database Queries | Network Calls |
|-----------|---------|------------------|---------------|
| Dashboard load | <100ms | 1 (cache) | 0 |
| Dashboard refresh | 2-5s | 1 (upsert) | N (brokers) |
| GTT create | 500ms-1s | 2 (insert + sync) | 2 (Zerodha) |
| GTT delete | 200-500ms | 1 (delete) | 1 (Zerodha) |
| Order cancel | 200-500ms | 1 (update) | 1 (Zerodha) |
| HMT trigger | 50-200ms | 1 (update) | 1 (Zerodha) |
| Realtime update | <50ms | 0 | 0 |

âœ… **Fast**: Instant loads from cache, sub-50ms updates

âœ… **Reliable**: Server-side execution, audit trail, retries
